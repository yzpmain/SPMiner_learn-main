# SPMiner 与 gSpan 对比分析

本文对比 SPMiner 和 gSpan 两类频繁子图挖掘方法，重点说明它们的核心思想、优缺点、适用场景，以及在工程落地时的取舍。

## 1. 基本结论

如果目标是“精确挖掘频繁子图”，gSpan 更像经典基线：它基于 DFS 编码和规范表示，结果定义清晰，理论上更偏向完备和准确。

如果目标是“在大规模图数据上高效搜索有用模式”，SPMiner 更像学习型方案：它借助图神经网络对候选 pattern 打分，再结合搜索策略筛选结构，通常更适合复杂数据、噪声数据和需要更强可扩展性的场景。

一句话概括：

- gSpan 偏“精确、经典、规则明确”。
- SPMiner 偏“学习、灵活、搜索驱动”。

## 2. 两种方法的核心机制

### 2.1 gSpan

gSpan 是经典的频繁子图挖掘算法，核心思路是：

1. 用 DFS 编码表示子图。
2. 通过规范序列避免重复枚举同构图。
3. 在搜索树上按频次支持度进行剪枝。
4. 逐步扩展频繁子图，输出满足最小支持度的 pattern。

它的特点是：结果完全由支持度阈值驱动，不依赖训练数据，也不需要学习模型。

### 2.2 SPMiner

SPMiner 是学习型频繁子图挖掘框架。它先训练一个子图匹配模型，把图编码到 embedding 空间中，然后在挖掘阶段使用这个模型为候选 pattern 打分，再通过贪心搜索或 MCTS 进行模式探索。

它的特点是：

- 不只依赖简单支持度统计。
- 可以借助神经网络捕捉更复杂的结构相似性。
- 更适合和下游任务结合，找“有用”而不只是“频繁”的模式。

## 3. 代码级验证

本仓库没有单独实现 gSpan，因此这里的“代码级比较”主要是两部分：

1. 对 SPMiner 的实际代码路径做验证。
2. 根据 gSpan 的标准算法流程，对比它在代码组织和执行方式上的差异。

### 3.1 SPMiner 的真实代码链路

从代码上看，SPMiner 的完整流程是清晰串联的：

| 环节 | 关键文件 | 代码层面看到的行为 |
| --- | --- | --- |
| 训练 / 加载模型 | [subgraph_mining/decoder.py](subgraph_mining/decoder.py) | 先根据 `method_type` 构造 `OrderEmbedder` 或 `BaselineMLP`，再从 `model_path` 加载 checkpoint |
| 图编码器 | [common/models.py](common/models.py) | `SkipLastGNN` 用消息传递层把图映射到 embedding；`OrderEmbedder.predict()` 输出子图违例分数 |
| 邻域采样 | [subgraph_mining/decoder.py](subgraph_mining/decoder.py) | 在 `pattern_growth()` 中从 `dataset` 采样 `neighs`，并通过 `utils.batch_nx_graphs()` 变成批量张量 |
| 候选扩展 | [subgraph_mining/search_agents.py](subgraph_mining/search_agents.py) | `GreedySearchAgent` 和 `MCTSSearchAgent` 都会枚举 frontier 候选并调用 `model.emb_model(...)` 计算新状态 embedding |
| 搜索策略 | [subgraph_mining/search_agents.py](subgraph_mining/search_agents.py) | Greedy 依赖排名得分，MCTS 依赖 visit count、UCT 和累计价值回传 |
| 输出结果 | [subgraph_mining/decoder.py](subgraph_mining/decoder.py) | 生成 pattern 图像并序列化到 `out_path` |

这说明 SPMiner 的实现不是“先枚举所有子图再过滤”，而是“先采样，再嵌入，再搜索”，代码结构上已经体现出学习驱动的性质。

### 3.2 从代码组织看，SPMiner 和 gSpan 的差异

如果把 gSpan 也按代码组织来对照，可以得到下面的差异：

| 对比点 | SPMiner 的代码表现 | gSpan 的典型代码表现 |
| --- | --- | --- |
| 核心入口 | 训练入口 + decoder 入口 | 单个挖掘入口即可 |
| 是否需要模型 | 需要 `torch.load()` 和 embedding 计算 | 不需要训练模型 |
| 候选生成 | 先采样邻域，再由搜索代理扩展 frontier | 直接按 DFS code 扩展频繁边 |
| 状态表示 | 图 embedding、WL hash、访问统计 | DFS 编码、规范序列、支持度计数 |
| 剪枝依据 | 模型打分 + 搜索统计 + frontier 约束 | 最小支持度 + 规范性剪枝 |
| 输出粒度 | pattern 图、图像、pickle 结果 | 频繁子图及其支持度 |

### 3.3 代码验证结论

从当前仓库代码可以直接确认：

- SPMiner 的实现是“先学习再挖掘”，而且搜索阶段强依赖 `common/models.py` 中的 embedding 和 `predict()`。
- `subgraph_mining/search_agents.py` 里的 Greedy 和 MCTS 都在使用图神经网络输出作为启发信号，而不是纯粹做组合枚举。
- 该仓库没有 gSpan 的源码实现，因此无法在本项目内直接运行 gSpan 作逐行对照；这里对 gSpan 的比较是基于其经典算法流程，而不是仓库内代码。

这也意味着：如果你要做“工程上的直接对比实验”，当前仓库更适合作为 SPMiner 一侧的实现基线；gSpan 需要额外引入独立实现或第三方库。

## 4. 对比维度

| 维度 | SPMiner | gSpan |
| --- | --- | --- |
| 方法范式 | 学习 + 搜索 | 规则驱动 + 枚举搜索 |
| 核心依据 | GNN 打分、模式搜索 | DFS 编码、支持度剪枝 |
| 是否需要训练 | 需要 | 不需要 |
| 结果可解释性 | 中等，依赖模型打分 | 较强，结果由支持度和规范编码决定 |
| 可扩展性 | 通常更适合大规模和复杂场景 | 数据量大时容易面临组合爆炸 |
| 对噪声的鲁棒性 | 取决于训练质量，通常更灵活 | 依赖严格的频次定义，面对噪声时可能不够灵活 |
| 输出目标 | 更偏“有用模式” | 更偏“频繁模式” |
| 调参成本 | 较高，需要训练和搜索参数 | 相对较低，主要是支持度和枚举参数 |

## 5. SPMiner 的优势

### 5.1 更适合复杂图结构

SPMiner 通过 GNN 学习结构表示，能够捕捉更高层的图模式，而不只是做纯粹的组合枚举。对于结构较复杂、属性较丰富、或者模式边界不那么清晰的数据，这种做法往往更有优势。

### 5.2 更容易结合下游任务

由于它本身就是学习框架的一部分，SPMiner 的输出模式可以和图匹配、图检索、图分类等任务衔接起来。也就是说，它不仅能挖“频繁”，还更容易挖到“对任务有意义”的结构。

### 5.3 搜索更灵活

SPMiner 使用 Greedy 或 MCTS 这类搜索策略，可以根据任务目标定制探索方向。相比纯枚举算法，这种策略在工程上更容易做剪枝、重排序和启发式优化。

### 5.4 对大图场景更友好

在实际项目中，完全枚举常常难以承受。SPMiner 把问题拆成采样、嵌入、打分、搜索几个步骤，通常更容易控制时间和资源消耗。

## 6. SPMiner 的劣势

### 6.1 依赖训练质量

SPMiner 的效果高度依赖子图匹配模型。如果编码器训练得不好，后续搜索得到的 pattern 质量也会受影响。也就是说，它不是“开箱即得”的方法。

### 6.2 结果稳定性弱于纯规则方法

同样的数据、不同的随机种子、不同的训练轮次，可能会得到不同的 pattern。相比 gSpan 这种以支持度为主的经典方法，SPMiner 的输出更依赖模型状态。

### 6.3 参数更多

它不仅要调挖掘参数，还要调匹配模型参数、搜索策略参数、采样参数。对于只想做标准频繁模式挖掘的场景，这种复杂度未必值得。

## 7. gSpan 的优势

### 7.1 经典且成熟

gSpan 是频繁子图挖掘领域的经典算法，思路清晰、实现路径明确，在很多教学、基准和论文场景里都很有代表性。

### 7.2 不需要训练

gSpan 不依赖神经网络，不需要额外的训练数据、checkpoint 或优化过程。只要给定图数据库和最小支持度，就可以直接运行。

### 7.3 结果定义明确

gSpan 的输出就是满足支持度阈值的频繁子图，解释起来非常直接。对于需要严格频次定义、报告可复现结果的任务，它的优势很明显。

### 7.4 理论与语义更稳定

由于它依赖规范表示和支持度约束，gSpan 的挖掘目标非常明确，不容易受到模型偏差影响。

## 8. gSpan 的劣势

### 8.1 枚举开销大

随着图规模和模式复杂度上升，gSpan 容易受到组合爆炸影响。即便有剪枝，面对大图、稠密图或者低支持度阈值时，计算量也可能很高。

### 8.2 对“有用模式”的表达有限

gSpan 主要优化的是频次，而不是任务相关性。换句话说，它擅长找“常见模式”，但不一定能找到“最有用的模式”。

### 8.3 难以结合学习框架

如果后续想把 pattern 和图表示学习、图检索、下游分类任务结合起来，gSpan 往往需要额外的后处理步骤。

## 9. 适用场景建议

### 适合用 gSpan 的情况

- 你要的是标准频繁子图挖掘。
- 你需要清晰、稳定、易解释的频繁模式结果。
- 你希望避免训练过程和模型不稳定性。
- 数据规模中等，或者支持度阈值不至于太低。

### 适合用 SPMiner 的情况

- 你要处理更复杂、更大规模的图数据。
- 你希望 pattern 不只是频繁，还要对下游任务有价值。
- 你接受“训练 + 搜索”的流程，并愿意用更多参数换灵活性。
- 你想把挖掘过程和图神经网络表示学习结合起来。

## 10. 从工程角度怎么看

如果把两者放在工程实践里比较，可以把它们看成两种不同目标的工具：

- gSpan 是“高确定性、低学习成本”的传统算法。
- SPMiner 是“更灵活、可学习、可扩展”的现代框架。

在很多项目里，两者并不是完全替代关系，而是互补关系：

- 先用 gSpan 做基线和频繁模式基准。
- 再用 SPMiner 找更贴近任务的高价值结构。

## 11. 结论

如果只追求经典频繁子图挖掘，gSpan 更直接、更稳定，也更容易解释。

如果追求更强的表达能力、任务相关性和大规模搜索能力，SPMiner 更有潜力，但代价是训练复杂度和结果波动更高。

因此，二者的优劣不是绝对的：

- gSpan 更适合“标准答案式”的频繁子图挖掘。
- SPMiner 更适合“学习驱动式”的模式发现。

在本仓库的语境下，SPMiner 的优势主要体现在把子图匹配与模式挖掘打通，而 gSpan 的优势则在于它仍然是一个非常强的、经典的对照基线。
